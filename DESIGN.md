## Problem Statement

传统的生物学习方式存在以下问题：

1. **答题反馈滞后**：学生完成练习后，无法立即获得针对具体答案的详细评审
2. **缺乏个性化指导**：标准答案往往只展示最终结果，不展示思考过程和优化路径
3. **术语使用不规范**：学生难以自行发现答题中的术语误用和逻辑漏洞
4. **难以系统性提升**：缺少对答题策略的持续优化和迭代机制

**核心需求**：构建一个能够模拟"学生做题-对照标准-发现偏差-修正思维"完整学习闭环的智能体系统。

---

## What Makes This Cool

这个系统的真正价值在于：

1. **自我进化的答题能力**：智能体能够从评分标准中提取模式，自动优化答题策略
2. **多角色协作反思**：模拟真实学习场景中的学生、老师、反思者三角关系
3. **精准的术语纠正**：特别针对生物学术表达的严格规范性进行优化
4. **透明的改进过程**：展示每一轮迭代的具体改进点和策略调整

**最令人惊喜的版本**：智能体在使用一段时间后，能够总结出特定学生常见错误的模式，并主动生成针对性的练习策略——就像一个真正经验丰富的生物老师一样。

---

## Constraints

1. **依赖外部LLM API**：需要Deepseek/Kimi等供应商的API key
2. **评分标准质量**：学习效果依赖于提供的评分标准文档的质量和完整性
3. **成本考虑**：多轮反思循环会增加API调用成本
4. **文档解析限制**：Word文档中的图片和复杂格式可能无法完全准确提取

---

## Premises

1. **LLM具备足够的推理能力**：Deepseek/Kimi能够理解复杂的生物学概念和评分逻辑
2. **评分标准蕴含可学习模式**：标准答案和评分细则中包含可提取的答题策略
3. **多轮迭代能够持续改进**：通过反思-修正循环，答案质量能够单调或近似单调提升
4. **术语规范性是可量化的**：生物学术语的使用准确性可以通过规则+模型的方式评估

### Approach B: LLM-based 反思循环（Recommended）

**总结**：利用 Deepseek/Kimi 的推理能力，构建一个多智能体系统：Actor（学生）生成答案，Reviewer（老师）对照评分标准评审，Reflector（反思者）分析差距并生成改进策略。

**工作量**：M（3-4 周）
**风险**：中

**优点**：
- 充分利用 LLM 的推理和语言理解能力
- 可以从评分标准中提取和学习模式
- 架构灵活，可扩展到不同题型

**缺点**：
- 依赖外部 API，成本和延迟需要考虑
- 需要精心设计的 prompt engineering
- 需要处理 LLM 的不确定性

**复用**：现有的 LLM 客户端代码模式

## Recommended Approach

**推荐选择 Approach B（LLM-based 反思循环）**。

### 理由：

1. **最符合核心目标**：能够直接实现"学生做题-对照标准-发现偏差-修正思维"的闭环
2. **工作量适中**：3-4 周可以完成核心功能，看到实际效果
3. **可扩展性强**：基础架构完成后，可以逐步添加 Approach C 的学习组件
4. **已有资源丰富**：你已经有了大量试题、评分标准和课本资料
5. **技术栈成熟**：Deepseek/Kimi 的 API 已经足够稳定，适合这种应用场景

## Open Questions

1. **API 预算**：你是否有预估的 API 调用预算？有。

2. **文档解析复杂度**：Word 文档中的图片（如实验图、流程图）对答题很重要，你希望如何处理这些图片？
   - **方案 A（推荐）**：使用本地 Ollama + DeepSeek OCR 将图片转换为 Markdown
     ```bash
     ollama run deepseek-ocr "/path/to/image" "<|grounding|>Convert the document to Markdown."
     ```
     - 优点：本地运行，保护隐私；无额外 API 成本；支持批量处理
     - 适用：实验图、流程图、表格等文档图片
   - 方案 B：使用多模态 LLM（如 GPT-4V、Claude 3V）处理
   - 方案 C：人工预处理，将图片描述为文本

3. **PDF文档支持**：项目需要处理PDF格式的生物学试题吗？如果需要，如何处理？
   - **方案 A（推荐）**：参考MarkPDFDown项目设计，使用PyMuPDF直接转换PDF页面为图片，支持页面范围选择
     - 优点：高效，支持大型文档，减少依赖（避免pdf2image）
     - 设计：工厂模式（FileWorker基类），统一接口，支持重试机制
   - 方案 B：使用现有pdf2image方案，保持向后兼容
   - 方案 C：仅支持Word格式，PDF需要用户手动转换

4. **开源协议**：你希望使用MIT
